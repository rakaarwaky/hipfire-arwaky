SPECIFICATION — HALYARD DISTRIBUTED CACHE, v3.

Overview. Halyard is a write-through, read-mostly cache layer deployed as a fleet of stateless front-end nodes ("edge nodes") backed by a consistent-hash ring of storage nodes ("shards"). Each edge node holds an in-memory LRU of recently requested entries; the shards hold the authoritative cached copies in a memory-mapped log-structured store. The relational database of record sits behind the shards and is never read directly by clients.

Request path. A client issues GET(key) to any edge node via a load balancer. The edge node first checks its local LRU. On a local hit, it returns immediately and asynchronously bumps the entry's recency. On a local miss, the edge node hashes the key onto the ring, forwards the request to the owning shard, and on the shard's response inserts the entry into its local LRU at the head, evicting the tail if the LRU is at capacity. On a shard miss, the shard reads the database of record, populates its log-structured store, and returns the value; the shard also records the key in a per-shard Bloom filter used to short-circuit future negative lookups.

Write path. A client issues PUT(key, value) to an edge node, which forwards it to the owning shard. The shard appends the new value to its log, updates an in-memory hash map from key to the latest log offset, and synchronously writes through to the database of record before acknowledging. After the database acknowledges, the shard publishes an invalidation message on a fan-out bus naming the mutated key. Every edge node subscribes to the bus and, on receiving an invalidation, drops the named key from its local LRU if present. The originating edge node does not special-case its own writes; it relies on the bus message like any other node.

Recency and eviction. Each edge node's LRU is a doubly linked list plus a hash map, capacity bounded by a configured byte budget. The shard's log-structured store is compacted by a background thread that rewrites live entries into a fresh segment and reclaims dead segments once their entries are superseded; compaction is paced to stay below a target write-amplification ratio. The Bloom filter is rebuilt during compaction and never deleted from incrementally, so a key removed from the database can still register as "possibly present" until the next compaction rebuild.

Consistency model. Halyard advertises read-your-writes consistency only for clients that pin to a single edge node for the duration of a session; clients that move between edge nodes may observe a stale value for the propagation window of the invalidation bus. The bus delivers at-least-once and is not ordered across keys. Invalidation messages carry only the key, not the new value or a version, so an edge node that receives a late or duplicated invalidation simply drops the key and re-fetches on the next read.

Failure handling. If a shard becomes unreachable, edge nodes serve any locally cached entries for that shard's key range and return errors for misses until the ring rebalances. Ring rebalancing reassigns the failed shard's key range to its successors; the successors cold-start with empty stores and repopulate from the database on demand. The invalidation bus is itself replicated; if an edge node disconnects from the bus, it flushes its entire LRU on reconnect because it cannot know which keys were invalidated while it was disconnected.

Operational notes. Edge nodes are autoscaled on request rate. A freshly started edge node begins with an empty LRU and warms organically. Shards are not autoscaled; adding a shard is a planned operation that triggers a controlled ring expansion and a background hand-off of the migrated key range. Metrics of interest include local hit rate, shard hit rate, invalidation propagation latency, compaction write-amplification, and the false-positive rate of the per-shard Bloom filters.
