# Relicense decision record: MIT → Apache 2.0 (May 2026)

**Date:** 2026-05-19
**Decided by:** Kaden Schutt (`@Kaden-Schutt`, `kaden@hipfire.dev`)
**Branch:** `relicense-apache2`
**Status:** Implemented in a single commit on the relicense branch;
pending owner review before merge.

## Summary

hipfire was originally MIT-licensed (LICENSE added in commit
`b068a2ac`, 2026-04-27). Effective with the relicense commit on
branch `relicense-apache2`, the canonical repository is relicensed
to **Apache License 2.0** with strong attribution practices:

- A per-Apache-2.0-§-4(d) `NOTICE` file at the repo root.
- Per-file SPDX headers on all in-repo first-party source code.
- An updated `CONTRIBUTING.md` that codifies the DCO sign-off
  requirement and downstream attribution obligations.
- A README + CREDITS pointer to this decision record.

## Motivation

The point of the relicense is **accreditation protection, not IP
protection**. We accept that downstream forks can use the code; we
want attribution to be durable and enforceable.

Triggering observation: a downstream fork called "hipEngine" exists
under AGPL 3.0. Because MIT is more permissive than AGPL, the fork
takes freely from hipfire but hipfire cannot freely pull from the
fork. The owner is frustrated by this asymmetry and, separately, by
the fork's weak preservation of attribution. Apache 2.0's § 4
imposes specific, copyright-grade obligations to retain copyright
and attribution notices when redistributing (`Jacobsen v. Katzer`,
535 F.3d 1373 (Fed. Cir. 2008)). Switching to Apache 2.0 + a strong
NOTICE + per-file SPDX headers gives the project an enforceable
attribution backbone where MIT's "include the LICENSE text" clause
was effectively toothless in practice.

## Legal basis

Kaden Schutt is the repository owner and the primary copyright
holder for the substantial majority of code in this repository. As
the copyright holder for his own work, Kaden is free to offer that
work under any license he chooses, including offering it under
Apache 2.0 alongside the prior MIT grant — MIT does not bar the
copyright holder from dual-licensing or relicensing his own
contributions.

For other contributors:
- Their MIT-licensed contributions remain MIT-licensed in perpetuity
  for anyone using existing MIT-tagged versions. That grant cannot
  be revoked.
- **MIT and Apache 2.0 are forward-compatible**: code contributed
  under MIT can be combined into an Apache 2.0 work, because MIT's
  conditions (preserve copyright + permission notice) are a strict
  subset of Apache 2.0's § 4 attribution requirements. (See
  <https://www.apache.org/legal/resolved.html#category-a> — Apache
  Software Foundation lists MIT/X11 as a Category A license, freely
  combinable into Apache-2.0 works.)
- Original contributor copyright notices MUST be preserved in any
  file they substantially contributed to (Apache 2.0 § 4(c)). The
  SPDX header sweep + `CREDITS.md` inventory together satisfy this.

### Header templates

**Sole author (Kaden):**

```
// SPDX-License-Identifier: Apache-2.0
// Copyright (c) 2026 Kaden Schutt
// hipfire — see LICENSE and NOTICE in the project root.
```

**With substantial secondary author (≥30% of source lines per
`git blame` at SPDX-sweep time):**

```
// SPDX-License-Identifier: Apache-2.0
// Copyright (c) 2026 Kaden Schutt
// Copyright (c) 2026 <Secondary Author Name>
// hipfire — see LICENSE and NOTICE in the project root.
```

Comment styles:

- Rust (`.rs`), HIP (`.hip`, `.cuh`): `// ...`
- Python (`.py`), shell (`.sh`): `# ...`, after the shebang. For
  Python with a shebang and a module-level docstring (the common
  case in `scripts/`), the SPDX block goes BETWEEN the shebang and
  the docstring, separated by a single blank line each side.

### Tooling

The sweep is performed by `scripts/governance/apply_spdx_headers.py`
(idempotent, ≥30%-blame secondary-author detection, `--dry-run` by
default, `--apply` to write). Re-running it on an already-headered
tree is a no-op except for files where a previously-secondary
contributor has crossed the 30% threshold for the first time.

The script is committed alongside the sweep so future contributors
have a single canonical way to (a) bring a new file into compliance
and (b) re-audit the tree after substantial refactors.

## Course correction 2026-05-19: dual licensing instead of unilateral relicense

**Status:** Implemented in a follow-up commit on `relicense-apache2`
after the initial unilateral-Apache-2.0 commit (`79167568`) landed.
History is NOT rewritten; the course correction is additive.

### What changed

After commit `79167568` landed, the licensing posture was reviewed
and changed from **unilateral Apache-2.0 relicensing of the entire
codebase** to **dual MIT / Apache-2.0 licensing** matching the Rust
ecosystem norm. The follow-up commit on this same branch implements
the new posture without rewriting any history.

Concretely:

- `LICENSE-MIT` was restored from the pre-relicense state
  (`d46f81b6:LICENSE`), preserving the original MIT text verbatim.
- The Apache-2.0 text moved from `LICENSE` to `LICENSE-APACHE`.
- `LICENSE` was replaced with a short dual-license pointer file
  naming both LICENSE-MIT and LICENSE-APACHE.
- `NOTICE`, `CONTRIBUTING.md`, `README.md`, and `CREDITS.md` were
  rewritten to describe the dual-license posture.
- A tag `v-mit-final` was placed on `d46f81b6` so the
  final-pure-MIT state has a stable name.

The per-file SPDX-License-Identifier headers were rewritten across
the 686 in-scope first-party source files using
`scripts/governance/apply_spdx_headers.py --rewrite-spdx --blame-rev
d46f81b6 --apply` (so the SPDX-administration commit's own added
lines don't skew authorship calculation). Result:

|  count | SPDX identifier        | meaning |
|-------:|-----------------------|---|
|    638 | `Apache-2.0`          | substantive authors entirely in {Kaden, Kevin} (opted into Apache-2.0 going forward) |
|     44 | `MIT`                 | substantive authors entirely outside {Kaden, Kevin} — preserves their original MIT grant rather than relicensing in absentia |
|      5 | `MIT OR Apache-2.0`   | mixed substantive authorship across the opt-in boundary — conservative dual tag until the non-K/K author opts in |

The 5 dual-licensed files are:


The per-file structured report is at
`/tmp/hipfire-spdx-correction-report.txt` (also embedded in the

### Why we changed it

The unilateral Apache-2.0 relicense was legally defensible (MIT and
Apache-2.0 are forward-compatible; the project owner has the right
to relicense his own MIT-granted code; MIT contributions from others
combine into an Apache-2.0 work because MIT's conditions are a
strict subset of Apache-2.0's § 4 attribution requirements). The
reasons we changed to dual instead are:

1. **Contributor agency.** Even though forward-compatibility makes
   the relicense legal, it puts existing contributors in a
   "consent-by-silence" position: their MIT grant gets recombined
   under Apache-2.0 without an explicit opt-in. Dual licensing
   instead lets each contributor pick — and for the substantial
   majority of contributors who haven't been asked, their files
   carry `SPDX-License-Identifier: MIT` until they opt in via a
   tracking issue.
2. **Rust ecosystem norms.** Most Rust crates (including the
   standard library, `tokio`, `serde`, `rayon`, etc.) ship under
   `MIT OR Apache-2.0`. New contributors arriving from the Rust
   ecosystem expect this posture and find a unilateral-Apache-only
   project surprising.
3. **Avoid putting existing contributors in a consent-by-silence
   position.** The 8 substantive non-Kaden/non-Kevin contributors
   identified in the contributor analysis above will be invited
   (via a tracking issue, opened separately by the maintainer
   after the branch is reviewed and merged) to opt in to dual
   licensing. Until they do, their solo files stay MIT and their
   shared files stay `MIT OR Apache-2.0` — never relicensed in
   absentia.

### What did NOT change

- The original unilateral-Apache-2.0 commit `79167568` stays in
  history (additive course correction, no force-push, no rewrite).
- The Apache-2.0 attribution machinery (NOTICE, § 4 obligations,
  Jacobsen-v-Katzer enforcement basis) still applies to all
  Apache-2.0-tagged files.
- Kaden's own copyright + Kaden+Kevin co-authored files remain
  Apache-2.0; the project's overall posture going forward is
  dual-licensed, and new contributions default to Apache-2.0 (via
  DCO sign-off in `CONTRIBUTING.md`).

### Follow-up items

1. **Open the contributor-opt-in tracking issue** on GitHub
   inviting the 8 substantive non-K/K contributors to opt their
   prior contributions into dual licensing. To be done manually by
   the maintainer after the branch is pushed and merged; CONTRIBUTING.md
   will be updated with the issue link at that time.
2. **`core.hooksPath` is now set to `.githooks`** in this worktree
   (per `git config core.hooksPath`), so the pre-existing
   `coherence-gate` + `speed-gate` pre-commit hook actually runs on
   future commits. Other clones of this repository will need to run

3. **New files added in this course correction:** `LICENSE-MIT`,
   `LICENSE-APACHE`, `AGENTS.md` (notice section prepended above
   the existing v0.1.9-alpha testing playbook), `PRIOR-ART.md`, and

### Reproducing the SPDX correction

To re-derive the per-file SPDX tags from the canonical
pre-relicense authorship (e.g. after a contributor opts in to dual
licensing and you want to refresh their files' tags):

```sh
# Refresh blame-based authorship -> SPDX classification across the
# full first-party scope. Updates the IDENTITY_MAP / KADEN_OR_KEVIN
# set in apply_spdx_headers.py first if a new contributor opted in.
python3 scripts/governance/apply_spdx_headers.py --rewrite-spdx \
    --blame-rev v-mit-final --apply \
    --report-path /tmp/hipfire-spdx-correction-report.txt
```

The tag `v-mit-final` points at the canonical pre-relicense commit
(`d46f81b6`). Using it as `--blame-rev` ensures the
SPDX-administration commits themselves never skew authorship
percentages.
